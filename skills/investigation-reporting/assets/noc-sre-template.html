Executive Summary

From 09:11Z to 09:58Z, roughly 18% of requests to shop.example.com failed with 502s or took seconds instead of milliseconds — about 12,400 degraded checkout requests. The cause was a single web-pool member, app-03, whose failing SSD made it progressively slower while lb01's connectivity-only health checks kept reporting it healthy and routing it traffic. The NOC was paged 3 minutes after onset; the fault was isolated to app-03 by 09:38Z and impact ended when the member was drained at 09:58Z. Service has run at baseline since. The SSD is being replaced; the durable fix is latency-aware health checking so a slow-but-alive member can never linger in the pool again.

Root Cause and Contributing Factors

Root CauseFailing SSD on app-03. Storage I/O stalls slowed request handling until responses took seconds or timed out, producing the 502s and latency users saw (E2, E4, E8).

TriggerSSD error rate crossed the stall threshold at ~09:11Z. SMART counters had been rising since 06:40Z — the failure announced itself 2.5 hours before impact, but those alerts route to the storage team's queue, not the NOC (E8).

ContributingLatency-blind health checks. lb01's connectivity-only check kept a member serving 9-second responses in rotation for the full 47 minutes. This is what turned a single-host fault into a service incident (E5).

ContributingNo per-member latency visibility on the service dashboard. Isolation took 24 minutes largely because the on-call had to assemble a per-member breakdown by hand (timeline, 09:14→09:38).

Hypotheses Considered

Bad deploy or config regression. The usual first suspect for sudden latency. Ruled out: change freeze in effect, no deploy or config-push traffic to the web tier in the window (E9).

Traffic surge or abusive client. Would explain pool-wide strain. Ruled out: request volume flat within ±4% of the 7-day baseline; no client concentration anomaly (E7).

Network congestion or packet loss. Would explain latency without host fault. Ruled out: retransmission and loss at baseline on all uplinks; the zero-window signal was localized to app-03, which indicates host backpressure rather than path trouble (E4, E7).

Database slowdown. Would degrade all members equally. Ruled out: DB-tier latency flat, and app-03's own queries ran at normal speed once requests reached the data layer (E3).
