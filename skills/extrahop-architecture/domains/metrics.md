# Metrics

Use this file for time-series analysis, dashboards, trend reasoning, broad pivots, and metric bucket behavior.

## What metrics are

Metrics are aggregated observations generated by sensors during real-time analysis. They can count events, measure timings, calculate rates, track distinct values, and retain top values over time.

Metrics are written to the packet sensor's local time-series datastore. Lookback depends on sensor sizing, traffic patterns, transaction rate, endpoint count, active protocols, and configuration.

## Why metrics matter

Metrics are built for fast exploration across large scope. They are the right first tool for:

- dashboards and charts;
- baseline and trend analysis;
- top-N and ranking workflows;
- broad historical pivots;
- detection inputs;
- narrowing a large question before querying records.

## Limits

Metrics only answer dimensions that exist as metrics. If a metric exists by URI, method, status code, or client, those dimensions are not automatically recombinable into arbitrary facets such as `URI x client x method`. Use records when arbitrary field correlation matters.

Custom metrics can be emitted by triggers.

## Metric buckets

Metrics are associated with sensor-local buckets:

- Devices: discovered network entities. L2 discovery anchors to MAC when possible; L3 and Remote L3 discovery anchor to IP where configured.
- Applications: user-defined service or business buckets, created through UI definitions or triggers.
- Networks: aggregate monitored network views. Packet sensors commonly expose VLAN-oriented network structure; flow telemetry commonly exposes flow senders and child interfaces.

Do not assume one global cross-sensor device table.

## Rollups

Metric granularity rolls up over time, for example from 30-second cycles to 5-minute and 1-hour views. Under pressure or finite storage limits, older metric history can age out.
